Windows EventLog Helpdesk

Version {{VERSION}} · {{RELEASE_DATE}}

This application converts live Windows event-log channels or offline EVTX/EVT files into complete Markdown evidence, performs local Ollama-based diagnostics, and keeps a follow-up conversation grounded in the loaded events and the latest analysis.

Recommended workflow

  1. Open Event Logs, refresh the tree, select a channel, and load it. Protected channels such as Security may require running the application as administrator.
  2. Set the event limit. A value of 0 / All reads the complete source; large logs may consume substantial memory and time.
  3. Optionally use a native Event Log XPath query, for example *[System[(Level=1 or Level=2 or Level=3)]].
  4. Review the table and complete event detail. The table search and severity filter do not alter the source data.
  5. Open Ollama Analysis, choose a model and answer language, then run the analysis.
  6. Use Talk About the System Events for grounded follow-up questions.

Data completeness

Every complete Markdown export contains, for every event, a normalized summary, the complete flattened XML field map, and the original raw XML. This is the evidence-preserving export. PDF reports are derived summaries and are not a substitute for the Markdown evidence file.

Ollama and non-blocking analysis

The default endpoint is http://127.0.0.1:11434. Model discovery uses /api/tags; analysis uses streamed Ollama requests in background workers. The interface remains usable and a progress bar remains visible until the answer arrives.

Auto mode uses a single request when the evidence fits the configured context budget. Larger evidence is split into diagnostic chunks, analyzed separately, and synthesized into a final report. Raw XML is excluded from the AI context by default because it expands context quickly, but it remains present in the complete Markdown export.

Diagnostic prompt behavior

The built-in prompt instructs the model to report only actionable faults, distinguish confirmed evidence from hypotheses, cite timestamps/providers/Event IDs/record IDs, avoid invented facts, prefer reversible remediation, and flag administrator, reboot, outage, registry, or data-loss implications. Event messages are treated as untrusted data so instructions embedded in logs are ignored.

System Context

The optional snapshot collects basic OS and hardware data, uptime, memory, disk usage, network addresses, reboot-pending markers, automatic services that are not running, and recent hotfixes. Review or edit the snapshot before sending it to Ollama. No snapshot is sent anywhere unless an Ollama analysis or chat request is made.

Privacy

With the default endpoint, event data remains on the local system and is sent only to the local Ollama service. If the endpoint is changed to another computer, the selected diagnostic context is sent to that host. The application does not use a cloud service by itself.

Reports and storage

Complete Markdown evidence is automatically saved after loading by default. A PDF report is automatically created after a successful analysis by default. Both behaviors and the output directory can be changed in Settings. PDF reports contain calculated severity/provider charts, tables, the Ollama diagnosis, and the optional system snapshot.

Windows text-to-speech

Enable TTS in Settings, choose a Windows SAPI voice, rate, and volume, then use Speak on the analysis or chat answer. Automatic speech after an analysis is optional and disabled by default.

Troubleshooting

ProblemLikely action
Security log cannot be openedRestart the application as administrator.
No Ollama models appearStart Ollama, verify the endpoint, run ollama list, then refresh models.
Analysis is very slowReduce the event limit, use warnings/errors scope, reduce raw XML context, or select a faster model.
Very large EVTX exportUse a filtered XPath or process a time/severity subset first. Complete conversion of huge logs can create very large Markdown files.
